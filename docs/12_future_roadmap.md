## 2. 近い将来（ハッカソン後 〜 3 ヶ月）

### 2.1 チャンネル用語集の自動生成

チャンネルの過去投稿から、プロジェクト固有の用語集を自動生成します。

```
機能:
  - チャンネル履歴を定期的に分析
  - 略語・専門用語・プロジェクト固有の言葉を抽出
  - 用語集を DynamoDB に保存
  - 補足文生成時に用語集を参照

技術:
  - EventBridge で定期実行
  - Bedrock で用語抽出
  - DynamoDB channel_contexts テーブルに保存
```

### 2.2 ユーザー別リテラシー学習

ユーザーのフィードバックや行動履歴から、リテラシーレベルを自動調整します。

```
機能:
  - フィードバック（「難しすぎる」「簡単すぎる」）を学習
  - ユーザーの役割・経験年数を自動推定
  - 補足レベルを自動調整

技術:
  - DynamoDB user_profiles テーブルを更新
  - フィードバックデータを分析
  - AgentCore Memory の活用
```

### 2.3 フィードバックによる品質改善

ユーザーのフィードバックを補足文生成の品質改善に活用します。

```
機能:
  - 「役に立った」「長すぎる」「違う」などのフィードバックを収集
  - フィードバックデータを分析
  - プロンプトの改善に活用

技術:
  - Slack のボタンアクション
  - DynamoDB feedback テーブル
  - 定期的なプロンプト改善サイクル
```

### 2.4 Drive / GitHub MCP の本格対応

MVP ではモックだった MCP 連携を本格実装します。

```
機能:
  - Google Drive から議事録・仕様書・設計メモを取得
  - GitHub から Issue・PR・README・ADR を取得
  - OAuth 認証の実装
  - 権限管理の実装

技術:
  - Google Drive MCP
  - GitHub MCP
  - OAuth 2.0
  - Secrets Manager でトークン管理
```

## 3. 中長期（3 ヶ月 〜 1 年）

### 3.1 Web 会議リアルタイム補足

Web 会議中に相手が分からなさそうな内容をリアルタイムで補足します。

```
ユースケース:
  - 会議中に専門用語が出たら、参加者の画面に補足を表示
  - 会議後に議事録の補足説明を自動生成
  - 会議中の質問を AI が先回りして補足

技術:
  - Amazon Chime SDK（音声・映像）
  - Amazon Transcribe（文字起こし）
  - リアルタイム補足生成
  - WebSocket / AppSync Subscription
```

### 3.2 ドキュメント作成支援

雑なメモから背景や前提を補完した仕様書・議事録を作成します。

```
ユースケース:
  - 箇条書きのメモから仕様書を生成
  - 会議の録音から議事録を生成
  - 設計メモから ADR（Architecture Decision Record）を生成

技術:
  - 本アプリの 4 Unit を拡張
  - Google Docs / Notion との連携
  - Drive MCP の活用
```

### 3.3 オンボーディング支援

新人が Slack の過去文脈を理解しやすくします。

```
ユースケース:
  - 新人が過去の投稿を読んで「これ何の話？」と思ったときに補足
  - チームの暗黙知を新人向けに説明
  - 過去の意思決定の背景を説明

技術:
  - 本アプリの拡張
  - チャンネル履歴の長期保存・検索
  - ユーザーオンボーディングフロー
```

### 3.4 A2A 対応（Agent の共通化・再利用）

文脈取得 Agent やリテラシーレビュー Agent を他アプリから再利用できるようにします。

```
A2A を使うケース:
  - 文脈取得 Agent を会議補足 Bot・ドキュメント作成 Bot から再利用
  - リテラシーレビュー Agent を提案書作成 Bot から再利用
  - GitHub 調査 Agent・Drive 調査 Agent を別チームが管理

技術:
  - Agent Card の定義
  - JSON-RPC over HTTP の実装
  - 認可・セッション管理
  - Strands Agents の A2A 対応

設計方針:
  - 本 MVP では Strands Orchestrator 内部で完結
  - 将来的に Agent を外部サービス化する段階で A2A を採用
  - A2A 対応を見据えた設計（Agent の責務を明確に分離）
```

## 4. エンタープライズ対応

### 4.1 SSO 連携

企業の SSO（Single Sign-On）と連携します。

```
機能:
  - SAML / OIDC による認証
  - 企業の ID 管理システムとの連携
  - ユーザー権限の自動同期

技術:
  - Amazon Cognito
  - SAML / OIDC
  - AWS IAM Identity Center
```

### 4.2 監査ログ

すべての操作を監査ログに記録します。

```
機能:
  - 誰がいつどの投稿を補足したかを記録
  - 補足文の内容を記録
  - 不審な操作を検知・アラート

技術:
  - CloudTrail
  - CloudWatch Logs
  - S3 への長期保存
```

### 4.3 権限連動検索

ユーザーの権限に基づいて、アクセスできる情報のみを検索します。

```
機能:
  - Drive / GitHub のアクセス権限を確認
  - 権限のない情報は検索・表示しない
  - チャンネルの参加者に合わせた補足文を生成

技術:
  - OAuth 2.0 スコープ管理
  - Drive / GitHub の権限 API
  - 動的な検索フィルタリング
```

## 7. 「人をダメにする」の深化

本アプリが普及すると、以下のような世界が実現します：

```
現在:
  発言者 → 丁寧に説明する努力 → 受け手が理解する

説明補足AI 普及後:
  発言者 → 雑に投稿する → AI が補足する → 受け手が理解する

さらに進むと:
  発言者 → 「あとはAIが補足してくれる」という前提で投稿する
  受け手 → 「AIが補足してくれるから追加質問しなくていい」という前提で読む
  
  → 人間の「説明する力」「文脈を読む力」が弱まる
  → でも、業務は回る
  → これが「人をダメにする」の本質
```

**キャッチコピー**: 「要点・背景・前提を整理しなくても、伝わってしまう世界へ」
