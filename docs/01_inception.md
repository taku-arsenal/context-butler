## 1. Intent（ビジネス意図）

### 一言で言うと

> **「雑に言っても、AIが伝わる説明にしてくれる」**

### 解決したいこと

業務チャット（Slack）における **説明不足・文脈不足・追加質問・認識合わせのコスト** を削減する。

同時に、発言者が背景や前提を丁寧に整理しなくても、AIが補足して伝わる状態を作ることで、ハッカソンテーマである **「人をダメにする」** を実現する。

### なぜ今これが必要か

- リモートワーク・非同期コミュニケーションの普及により、Slack 上の短い投稿が増加している
- 「これ何の話？」「背景は？」「何をすればいい？」という追加質問が繰り返されている
- 投稿者は毎回丁寧に説明するコストを払えない
- 新人・途中参加者・非エンジニアは文脈を追えない
- チームの暗黙知が多く、外部の人間には理解できない投稿が多い

### ハッカソンテーマとの接続

通常、人は他者に伝わるように説明するために以下の努力をしています：

| 人間がやっていること | 本アプリが代わりにやること |
|---------------------|--------------------------|
| 背景を整理する | 省略抽出 Agent が背景の欠如を検出し、文脈取得 Agent が補完する |
| 前提条件を書く | 補足文生成 Agent が前提を明示する |
| 関連資料を探す | 文脈取得 Agent が KB・Drive・GitHub から自動収集する |
| 専門用語を噛み砕く | 補足文生成 Agent が用語説明を追加する |
| 聞き手の知識レベルを考慮する | リテラシーレビュー Agent が読み手に合わせて調整する |
| 誤解が起きないように書く | リテラシーレビュー Agent が事実と推測を分離する |
| 必要な行動を明示する | 補足文生成 Agent が次アクションを整理する |

**結果**: 発言者は説明を考え込まず雑に投稿でき、聞き手は追加質問を減らせる。  
**人間の「説明する力」を弱らせ、人をダメにする** — しかし業務効率化としては普通に便利で、実用性もある。

## 2. User（対象ユーザー）

### プライマリユーザー

| ユーザー | 課題 |
|----------|------|
| Slack を使う開発チーム | 技術的な背景を省いた投稿が多く、追加質問が頻発する |
| Slack を使う営業・企画・情シスチーム | 専門用語や略語が多く、非エンジニアが理解できない |
| プロジェクトに途中参加した人 | 過去の経緯や意思決定の背景が分からない |
| 新人 | チームの暗黙知や用語が分からない |
| 非エンジニアのステークホルダー | 技術的な投稿の意味が分からない |

### セカンダリユーザー

| ユーザー | 課題 |
|----------|------|
| 説明が雑な人 | 丁寧に説明する時間がない、または説明が苦手 |
| 説明不足な投稿に困る人 | 追加質問するのが気まずい、または時間がない |

### ユーザーストーリー

```
As a 途中参加のプロジェクトメンバー,
I want to 分かりにくい Slack 投稿の三点リーダから補足説明を呼び出せる,
So that 追加質問せずに投稿の背景・前提・次アクションを理解できる.
```

```
As a 説明が雑な投稿者,
I want to 雑に投稿しても AI が補足してくれる,
So that 毎回丁寧に背景を書く手間を省ける.
```

## 3. Problem（解決する課題）

### 課題の構造

```
┌─────────────────────────────────────────────────────────────┐
│                    業務チャットの現状                         │
│                                                             │
│  投稿者                          受け手                      │
│  ┌──────────────────┐           ┌──────────────────────┐   │
│  │ 背景を省略して投稿 │  ──▶    │ 「これ何の話？」      │   │
│  │ 前提を書かない    │           │ 「背景は？」          │   │
│  │ 略語を使う        │           │ 「何をすればいい？」  │   │
│  │ 判断理由を省く    │           │ 「関連資料は？」      │   │
│  └──────────────────┘           └──────────────────────┘   │
│                                          │                  │
│                                          ▼                  │
│                              追加質問・認識合わせのコスト     │
│                              文脈を探す時間のロス            │
│                              新人・途中参加者の疎外感        │
└─────────────────────────────────────────────────────────────┘
```

### 具体的な問題

1. **Slack 投稿が短く、背景が不足している**
   - 例：「例の認証の件、来週から切り替える方向で。影響あるところだけ確認お願いします。」
   - 「例の認証」が何か、なぜ切り替えるのか、誰が何を確認すべきかが不明

2. **チーム内の暗黙知が多い**
   - 長期メンバーには自明でも、新人・途中参加者には分からない
   - 略語・プロジェクト固有の用語が説明なく使われる

3. **関連資料や Issue を探すのが大変**
   - 「どの Issue の話？」「どの議事録で決まった？」を自分で探す必要がある

4. **聞き手が追加質問しないと理解できない**
   - 追加質問するのが気まずい場合もある
   - 非同期コミュニケーションでは返答待ちが発生する

5. **投稿者が毎回丁寧に説明するのが面倒**
   - 忙しい中で背景・前提・次アクションを整理して書く時間がない

## 4. Solution（解決策）

### アプリの動作

```
┌─────────────────────────────────────────────────────────────┐
│                    説明補足AI の動作                          │
│                                                             │
│  1. ユーザーが分かりにくい投稿の三点リーダを押す             │
│  2. 「Explain with 説明補足AI」を選択                        │
│  3. アプリが 3 秒以内に受付確認を返す                        │
│  4. バックグラウンドで 4つの Agent ステージが実行される     │
│  5. 元投稿のスレッドに補足説明が返信される                   │
└─────────────────────────────────────────────────────────────┘
```

### 補足文の構成

生成される補足文には以下が含まれます：

| 要素 | 内容 |
|------|------|
| この投稿の要点 | 何についての投稿か |
| 背景 | なぜこの話が出ているか |
| 前提 | 知っておくべき前提条件 |
| 用語説明 | 略語・専門用語の説明 |
| 判断理由 | なぜこの判断になったか |
| 関連資料 / Issue | 参照すべき資料・Issue |
| 次に取るべき行動 | 誰が何をすべきか |
| 不確かな点 | この投稿だけでは不明な点 |

### 技術的アプローチ

Slack の **Message Shortcut**（三点リーダメニュー）を起点に、単一の Strands Orchestrator が 4 つの専門 Agent を順次制御する構成。

```
Message Shortcut
      │
      ▼
API Gateway → Lambda Ack（3秒以内）→ SQS FIFO
                                          │
                                          ▼
                                    Worker Lambda
                                          │
                                          ▼
                              AgentCore Runtime
                              Strands Orchestrator
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
              Agent Stage 1   Agent Stage 2   Agent Stage 3 → Stage 4
              省略抽出        文脈取得        補足文生成    レビュー
```

## 5. Unit 分解（AI-DLC の核心）

AI-DLC における Unit of Work は、Agent の種類ではなく **並行開発可能な作業単位**です。本アプリでは、MVPを作るための実装単位として以下の 4 Unit に分解します。

| Unit of Work | 目的 | 主な成果物 |
|--------------|------|------------|
| A. Slack起動・非同期ジョブ基盤 | Slack Message Shortcut から job を受け付け、3秒以内に ack し、非同期処理へ渡す | Slack App設定、API Gateway、Ack Lambda、SQS FIFO、Worker Lambda |
| B. AgentCore + Strands 補足生成パイプライン | 対象投稿と文脈から補足文を生成し、レビューして返す | AgentCore Runtime、Strands Orchestrator、4つのAgentステージ、プロンプト |
| C. ナレッジ・MCP連携 | KB、Google Drive、GitHub から補足に必要な根拠を取得する | Bedrock Knowledge Bases、AgentCore Gateway、Drive/GitHub MCP接続 |
| D. データ永続化・安全性・評価 | job履歴、権限制御、安全性、評価指標を整える | DynamoDB、S3、Guardrails方針、想定補足ポイント評価 |

実行時の AI 処理は、Unit B の中で以下の 4 つの Agent ステージとして実装します。

| Agentステージ | 役割 |
|---------------|------|
| 省略抽出 Agent | 対象投稿から省略・暗黙知・聞き手が詰まりそうな点を抽出する |
| 文脈取得 Agent | スレッド、KB、Drive、GitHub から必要文脈を収集・整理する |
| 補足文生成 Agent | 背景・前提・用語・判断理由・次アクションを整理した補足文を生成する |
| リテラシーレビュー Agent | 事実/推測の分離、補足過多、品質、安全性を確認する |

## 6. MVP スコープ

### Must（必須）

- [ ] Slack Message Shortcut から起動
- [ ] Slack リクエストを 3 秒以内に ack
- [ ] SQS で非同期化
- [ ] 対象メッセージを取得
- [ ] スレッド履歴を取得
- [ ] 省略抽出 Agent の実装
- [ ] 文脈取得 Agent の最小実装（Slack スレッド + 社内ナレッジ KB）
- [ ] 補足文生成 Agent の実装
- [ ] リテラシーレビュー Agent の実装
- [ ] AgentCore Runtime + Strands Orchestrator で 4 Agent ステージを順次実行
- [ ] 元投稿のスレッドに補足文を返信
- [ ] DynamoDB に job 履歴を保存
- [ ] README と AI-DLC Inception ドキュメントを整備

### Should（できれば）

- [ ] 文脈取得 Agent の強化（チャンネル履歴要約・Drive/GitHub）
- [ ] チャンネル履歴要約の利用
- [ ] ユーザーリテラシー設定
- [ ] Slack モーダルで補足レベル・想定読者を選択
- [ ] Bedrock Guardrails の利用

### Could（余力があれば）

- [ ] Google Drive MCP
- [ ] GitHub MCP
- [ ] フィードバックボタン

### Won't for MVP

- LINE 対応
- Web 会議リアルタイム対応
- A2A によるエージェント間通信
- 全社横断の大規模検索

## 8. Risks & Mitigations（リスクと対策）

| リスク | 影響 | 対策 |
|--------|------|------|
| 事実誤認（AIが誤った背景を生成） | 高 | リテラシーレビュー Agent で事実と推測を分離。不明点は「不明」と明記 |
| 過剰補足（長すぎる補足文） | 中 | リテラシーレビュー Agent で長さチェック。補足レベル設定で制御 |
| 機密情報の露出 | 高 | MVP は権限のある参加者のみを招待した Slack Private チャンネルで実施。Drive/GitHub 取得情報は要約し、直接貼り付けを禁止 |
| Slack API 権限不足 | 中 | 最小権限スコープで設計。必要スコープを事前に確認 |
| MCP 接続の複雑化 | 中 | MVP では Drive/GitHub のみに限定。モックでも可 |
| 生成遅延（30 秒超） | 中 | SQS 非同期化。「生成中です」メッセージで UX を補完 |
| 出力が長すぎる | 中 | リテラシーレビュー Agent でトークン数・文字数チェック |
| 「ただの要約 Bot」に見える | 低 | 省略抽出と文脈取得を強調。KB・Drive・GitHub 連携を見せる |
| A2A 実装コストの増大 | 低 | MVP では A2A を使わない。Strands Orchestrator 内部で完結させる |
| AgentCore Runtime 設定の遅延 | 中 | MVP から AgentCore 利用を第一候補にする。予選直前に詰まる場合のみ、同じ Agent 入出力契約を保った Bedrock 直接呼び出しで継続する |

## 9. 技術選定の判断理由

### Slack Message Shortcut を採用した理由

- 対象メッセージが明確で、業務 UX として自然
- メンション型よりチャンネルを汚しにくい
- 「この投稿を補足してほしい」という意図が明確
- Slack 公式の体験として馴染みやすい

### A2A を MVP で使わない理由

本 MVP では、4 つの Agent は同一 Strands Orchestrator 内部の処理単位であり、独立した外部 Agent 同士が通信する必要はありません。A2A を使うと Agent Card・JSON-RPC over HTTP・認可・セッション管理・タイムアウト・トレースなどの実装・説明コストが増えます。

MVP では実装複雑性を抑えつつ、将来的なマルチエージェント拡張性を確保する設計判断として、A2A は将来対応とします。

> 将来 A2A を使うケース：文脈取得 Agent を他アプリから再利用したい場合、GitHub 調査 Agent・Drive 調査 Agent・リテラシーレビュー Agent を別チームが管理する場合など。

### MCP を Drive / GitHub のみに限定した理由

MCP は「外部データソースに接続するための標準プロトコル」として限定利用します。Slack 文脈取得は Slack API を直接使うことで、MCP の接続複雑性を避けます。

## 11. 3 案比較と最終判断

当初は以下 3 案を検討しました：

| 案 | 概要 | テーマ適合性 | 技術アピール | 実装難易度 | 判断 |
|----|------|-------------|-------------|-----------|------|
| 案1: 朝活代行 | 朝活失敗時に「やった風」の画像・動画を生成 | ◎ | △ | 中 | 不採用 |
| **案2: 説明補足AI** | **Slack の説明不足投稿を AI が補足** | **○** | **◎** | **中〜高** | **採用** |
| 案3: 遅刻の言い訳 | 遅刻時に謝罪文・到着予測を生成 | △ | ○ | 高 | 不採用 |

**案2 を採用した理由**：

- 書類審査の「Intent の明確さ」「Unit 分解の適切さ」「ドキュメント品質」を最も出しやすい
- 業務課題としてのリアリティが高く、審査員に刺さりやすい
- AWS サービス（Bedrock・AgentCore・Knowledge Bases・MCP）を自然に活用できる
- AI-DLC の Unit of Work と実行時 Agent ステージを分けて説明しやすい
- 「人をダメにする」テーマとの接続も明確（説明責任の外注）
