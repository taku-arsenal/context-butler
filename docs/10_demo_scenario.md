# デモシナリオ

**プロジェクト名**: 説明補足AI（Explain Bot）  
**バージョン**: 1.0.0  
**作成日**: 2026-05-10

---

## 1. デモの目的

ハッカソン審査員に以下を伝えることがデモの目的です：

1. **「雑な投稿でも伝わってしまう」** ことを体感させる
2. **4 Unit の AI 処理** が実際に動いていることを見せる
3. **業務課題としてのリアリティ** を感じさせる
4. **「人をダメにする」テーマ** との接続を印象付ける

---

## 2. デモ用の悪い投稿例

### パターン 1: 認証切り替えの共有（メイン）

```
例の認証の件、来週から切り替える方向で。影響あるところだけ確認お願いします。
```

**この投稿に不足している情報:**

| 不足情報 | 説明 |
|---------|------|
| 「例の認証」とは何か | 独自認証？Cognito？OAuth？ |
| どのシステムの話か | 社内ポータル？API？全システム？ |
| なぜ切り替えるのか | セキュリティ？コスト？運用負荷？ |
| いつ決まったのか | 先週の定例？先月の会議？ |
| 誰が何を確認すべきか | 全員？特定の担当者？ |
| 影響範囲は何か | ログイン処理？ユーザー管理？権限チェック？ |
| 関連する Issue / 資料はどこか | GitHub Issue？議事録？仕様書？ |
| 確認後のアクションは何か | Issue に追記？スレッドに返信？ |

### パターン 2: 仕様変更の通知

```
通知の件、変更します。詳細は後で共有します。
```

### パターン 3: 障害報告

```
さっきのエラー、原因分かりました。対応します。
```

### パターン 4: 依頼

```
例のやつ、今週中にお願いします。
```

## 3. デモ用の準備

### 3.1 デモ用 Slack チャンネルの準備

1. 権限のある参加者のみを招待したデモ用 Private チャンネル `#demo-explain-bot` を作成
2. 以下のメッセージを事前に投稿しておく（または当日投稿）：
   ```
   例の認証の件、来週から切り替える方向で。影響あるところだけ確認お願いします。
   ```
3. 説明補足AI Bot をチャンネルに招待する

MVP では、この Private チャンネルを情報共有の境界として扱います。デモ用 KB / GitHub / Drive 相当データも、このチャンネル参加者が見てよい内容に限定します。

### 3.1.1 評価用の想定補足ポイント

デモ後に、生成結果が以下の期待補足ポイントをどれだけ満たしたかを確認します。MVP の定量評価では、8 項目中 7 項目程度、つまり 80% 以上を目標にします。

| # | 想定補足ポイント |
|---|----------------|
| 1 | 「例の認証」が社内ポータルの認証方式変更を指すこと |
| 2 | 独自認証から Cognito 連携へ切り替える話であること |
| 3 | まず開発環境が対象で、本番適用日は未確定であること |
| 4 | 背景にセキュリティ運用負荷や監査対応があること |
| 5 | 影響範囲がログイン処理・ユーザー管理・権限チェックであること |
| 6 | 関連 Issue / 議事録 / KB など根拠ソースに触れていること |
| 7 | 担当者が影響範囲を確認し、懸念点をスレッドへ返すべきこと |
| 8 | 本番適用日やロールバック手順など不明点を断定しないこと |

### 3.2 デモ用 Knowledge Base の準備

`demo/kb_docs/` に以下の Markdown ファイルを配置し、Bedrock Knowledge Bases に登録する：

**`demo/kb_docs/auth-migration.md`**:
```markdown
# 認証基盤移行プロジェクト

## 概要
社内ポータルの認証方式を独自認証から AWS Cognito 連携に移行するプロジェクト。

## 背景
- 現在の独自認証は運用負荷が高い
- セキュリティ監査で改善が求められている
- AWS 側に認証基盤を寄せることでコスト削減が見込める

## 移行方針
1. まず開発環境で Cognito 連携を試験導入
2. 問題なければステージング環境に適用
3. 本番環境への適用は別途判断

## 影響範囲
- ログイン処理
- ユーザー管理
- 権限チェック

## 関連 Issue
- GitHub Issue #123: 認証方式を Cognito に移行する
```

```markdown
# プロジェクト Alpha 用語集

## Cognito
AWS の認証サービス。ユーザー管理・認証・認可を提供する。

```

## 4. デモの流れ

### Step 1: 問題提起（30 秒）

```
「業務チャットでよくある光景を見てください。」

[Slack を表示]
「例の認証の件、来週から切り替える方向で。影響あるところだけ確認お願いします。」

「この投稿、何が問題か分かりますか？
 - 『例の認証』って何？
 - どのシステムの話？
 - 誰が何を確認すればいい？
 - 関連する資料はどこ？
 
 こういう投稿、毎日見ませんか？」
```

### Step 2: アプリの起動（15 秒）

```
「では、説明補足AI を使ってみます。」

[投稿の三点リーダをクリック]
[「Explain with 説明補足AI」を選択]

「これだけです。あとは AI が全部やってくれます。」
```

### Step 3: 処理中の説明（30 秒）

```
「今、バックグラウンドで 4 つの AI Agent が動いています。」

[処理ログを表示しながら]

「Unit 1: 省略抽出 Agent が、この投稿の何が足りないかを分析しています。
 Unit 2: 文脈取得 Agent が、スレッド・ナレッジベース・GitHub から関連情報を収集しています。
 Unit 3: 補足文生成 Agent が、背景・前提・次アクションを整理した補足文を生成しています。
 Unit 4: リテラシーレビュー Agent が、品質と安全性を確認しています。」
```

### Step 4: 補足文の確認（45 秒）

```
[Slack のスレッドに補足文が返信される]

「見てください。スレッドに補足文が返信されました。」

[補足文を読み上げる]

「補足です。この投稿は、社内ポータルの認証方式を既存の独自認証から
 Cognito 連携に切り替える件についての共有です。

 背景として、先週の定例でセキュリティ運用負荷を下げるため、
 認証基盤を AWS 側に寄せる方針が合意されています。
 Issue #123 でも認証遅延と確認漏れが課題として挙がっています。

 確認すべき人は、ログイン処理・ユーザー管理・権限チェックに
 関わる実装を持つ担当者です。

 次のアクションは、影響範囲を確認し、懸念点があれば来週の切り替え前に
 このスレッドへ返信することです。

 なお、本番適用の具体的な日程とロールバック手順は、
 この投稿とスレッドだけでは明記されていません。」

「追加質問、必要ですか？ 必要ないですよね。」
```

### Step 5: テーマとの接続（30 秒）

```
「これが『人をダメにする』です。

 発言者は雑に投稿しても、AI が背景や前提を補ってくれます。
 聞き手も、追加質問や文脈を探す努力を減らせます。

 説明責任と文脈整理を AI に外注できる時代が来ました。

 でも、業務効率化としては普通に便利です。
 これが説明補足AI です。」
```

## 5. デモ用の良い出力例

### パターン 1: 認証切り替えの共有

```
補足です。この投稿は、社内ポータルの認証方式を既存の独自認証から Cognito 連携に切り替える件についての共有です。

**背景**
先週の定例（2026-05-03）でセキュリティ運用負荷を下げるため、認証基盤を AWS 側に寄せる方針が合意されています。Issue #123 でも認証遅延と確認漏れが課題として挙がっています。

**対象範囲**
今回の変更対象はまず開発環境です。本番切り替えは別途判断予定です。利用者向け通知ではなく、ログイン処理・ユーザー管理・権限チェックに関わる内部実装が対象です。

**確認すべき人**
ログイン処理・ユーザー管理・権限チェックに関わる実装を持つ担当者です。

**次に取るべき行動**
影響範囲を確認し、懸念点があれば来週の切り替え前にこのスレッドへ返信してください。問題なければ Issue #123 に影響範囲を追記してください。

**不確かな点**
本番適用の具体的な日程とロールバック手順は、この投稿とスレッドだけでは明記されていません。
```

## 6. AI-DLC プロセスの見せ方

デモでは、AI-DLC の Unit 分解を視覚的に見せることで、審査員に「ただの要約 Bot ではない」ことを伝えます。

### Unit 1 の出力を見せる

```json
{
  "message_intent": "認証方式の切り替えに関する共有・依頼",
  "omitted_points": [
    "「例の認証」が何を指すか明示されていない",
    "対象システム名が明示されていない",
    "なぜ切り替えが必要なのかの背景・理由が不足している",
    "確認すべき人が明示されていない"
  ],
  "recommended_retrieval_plan": ["thread", "channel_summary", "github", "kb"]
}
```

### Unit 2 の出力を見せる

```json
{
  "retrieved_context": {
    "kb_context": [{"title": "認証基盤移行プロジェクト", "summary": "..."}],
    "github_context": [{"title": "Issue #123: 認証方式を Cognito に移行する", "summary": "..."}]
  },
  "missing_context": ["本番適用の具体的な日程は不明"]

```


## 7. デモ本番手順

### 事前確認（デモ 30 分前）

- [ ] Slack チャンネルにデモ用投稿が準備されている
- [ ] 説明補足AI Bot がチャンネルに招待されている
- [ ] AWS リソースが正常に動作している（Lambda・SQS・DynamoDB・Bedrock）
- [ ] Knowledge Bases にデモ用資料が登録されている
- [ ] CloudWatch Logs でエラーがないことを確認

### デモ当日の手順

1. Slack を開き、デモ用チャンネルを表示
2. デモ用投稿を表示（または当日投稿）
3. 三点リーダから「Explain with 説明補足AI」を選択
4. 処理中の説明をしながら待つ（約 15〜30 秒）
5. スレッドに補足文が返信されたことを確認
6. 補足文を読み上げ・解説
7. Unit 分解の処理ログを見せる

### バックアップ手順（デモが失敗した場合）

- 事前に生成した補足文のスクリーンショットを用意しておく
- 「デモ環境の都合で事前に生成した結果を見せます」と説明する
- Unit 分解の出力 JSON を手動で見せる

## 8. プレゼンとデモの時間配分

| セクション | 時間 | 内容 |
|-----------|------|------|
| 問題提起 | 1 分 | 業務チャットの課題を説明 |
| アプリ概要 | 1 分 | 何ができるかを説明 |

| 技術説明 | 2 分 | 4 Unit・AWS 構成を説明 |

| 将来展望 | 1 分 | A2A・LINE 対応などの展望 |
| まとめ | 30 秒 | キャッチコピーで締める |

**合計: 約 8 分 30 秒**（予選の発表時間に合わせて調整）
