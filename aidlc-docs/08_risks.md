# 08 Risks & Mitigations

**プロジェクト名**: Context Butler / 説明補足AI（Explain Bot）

---

## リスク一覧

### R-01: 事実誤認（AI が誤った背景を生成する）

| 項目 | 内容 |
|------|------|
| **影響** | 高 |
| **発生確率** | 中 |
| **説明** | 文脈取得 Agent が取得した情報が不十分・不正確な場合、補足文生成 Agent が誤った背景・前提を断定形で生成する可能性がある |
| **Mitigation** | リテラシーレビュー Agent で事実と推測を分離する。取得した文脈から確認できることのみ断定形で書き、確認できないことは「〜と考えられます」と書く。不明点は「この投稿だけでは明記されていません」と明示する。Bedrock Guardrails で根拠なし断定を抑制する（Should） |
| **残存リスク** | KB・Drive・GitHub に誤った情報が含まれている場合は検出できない |

---

### R-02: 過剰補足（長すぎる・不要な情報が多い補足文）

| 項目 | 内容 |
|------|------|
| **影響** | 中 |
| **発生確率** | 中 |
| **説明** | 補足文生成 Agent が必要以上に多くの情報を含めた長い補足文を生成し、受け手が読む気を失う |
| **Mitigation** | リテラシーレビュー Agent で補足過多をチェックし、200〜500 文字程度に収める。省略抽出 Agent の `recommended_retrieval_plan` で必要な情報源のみ参照し、不要な検索をしない。補足レベル設定（かんたん / 標準 / 詳細）で制御できるようにする（Should） |
| **残存リスク** | 補足レベルが MVP では固定のため、ユーザーが調整できない |

---

### R-03: 機密情報の過剰露出

| 項目 | 内容 |
|------|------|
| **影響** | 高 |
| **発生確率** | 低（MVP の権限境界で制御） |
| **説明** | Drive / GitHub から取得した機密情報がそのまま Slack スレッドに投稿される。または、Slack 投稿に含まれる個人情報・機密情報が補足文に過剰に含まれる |
| **Mitigation** | MVP は権限のある参加者のみを招待した Slack Private チャンネルで実施する。Drive / GitHub 取得情報は要約・抽象化して使用し、そのまま貼らない。リテラシーレビュー Agent が個人情報（氏名・メールアドレス・電話番号）の出力をチェックする。Bedrock Guardrails で個人情報・機密情報の過剰出力を抑制する（Should）。DynamoDB の TTL で Slack 本文を 30 日で自動削除する |
| **残存リスク** | Guardrails が MVP では Should のため、リテラシーレビュー Agent のみに依存する期間がある |

---

### R-04: Slack API 権限不足

| 項目 | 内容 |
|------|------|
| **影響** | 中 |
| **発生確率** | 低 |
| **説明** | Slack Bot に必要なスコープが付与されていない場合、メッセージ取得・スレッド返信が失敗する |
| **Mitigation** | 必要スコープ（`chat:write` / `groups:history` / `users:read`）を事前に確認し、Slack App 設定時に付与する。MVP は Private チャンネルのみを対象にするため、`channels:history` は不要 |
| **残存リスク** | Slack の権限モデル変更により、必要スコープが変わる可能性がある |

---

### R-05: MCP 接続の複雑化

| 項目 | 内容 |
|------|------|
| **影響** | 中 |
| **発生確率** | 中 |
| **説明** | Google Drive MCP・GitHub MCP の OAuth 設定・AgentCore Gateway の設定が複雑で、MVP 期間内に完成しない |
| **Mitigation** | Drive / GitHub MCP はMVPで実装を目指すが、期間内に安定化できない場合はFutureとして扱う。文脈取得 Agent の外部呼び出しをインターフェース化し、Bedrock Knowledge Bases（デモ用 Markdown KB）を Must として先に実装し、MCP なしでもデモできる状態を確保する |
| **残存リスク** | MCP なしのデモでは「ただの KB 検索 Bot」に見えるリスクがある |

---

### R-06: 生成遅延（30 秒超）

| 項目 | 内容 |
|------|------|
| **影響** | 中 |
| **発生確率** | 中 |
| **説明** | Bedrock の推論時間・KB 検索・MCP 呼び出しが重なり、補足文生成が 30 秒を超える |
| **Mitigation** | 軽い Agent（省略抽出・文脈取得）は Claude 3.5 Haiku / Amazon Nova Lite で高速化する。SQS 非同期化により Slack の 3 秒制約は回避済み。「補足を生成中です」メッセージで UX を補完する。MCP 呼び出しは `recommended_retrieval_plan` に基づき必要な場合のみ実行する |
| **残存リスク** | Bedrock のスロットリングが発生した場合、遅延が増加する |

---

### R-07: 「ただの要約 Bot」に見える

| 項目 | 内容 |
|------|------|
| **影響** | 中 |
| **発生確率** | 中 |
| **説明** | 審査員・デモ参加者が「これは単に投稿を要約しているだけでは？」と感じ、AI-DLC の価値・AgentCore の活用・MCP 連携の意義が伝わらない |
| **Mitigation** | デモで省略抽出 Agent の構造化 JSON 出力（何が省略されているかの分析）を見せる。KB・Drive・GitHub から取得した根拠を補足文に反映させ、「外部文脈を取得して補足している」ことを示す。リテラシーレビュー Agent による事実/推測の分離・品質チェックを説明する。AgentCore Runtime + Strands Orchestrator の 4 Agent ステージ処理ログを見せる |
| **残存リスク** | デモシナリオが弱い場合、差別化が伝わりにくい |

---

### R-08: AgentCore Runtime 設定の遅延

| 項目 | 内容 |
|------|------|
| **影響** | 中 |
| **発生確率** | 中 |
| **説明** | AgentCore Runtime の設定・デプロイが予選直前に詰まり、MVP が完成しない |
| **Mitigation** | MVP から AgentCore Runtime の利用を第一候補にし、早期に設定を開始する。詰まった場合のみ、同じ Agent 入出力契約（JSON スキーマ）を保った Bedrock 直接呼び出しで継続し、予選後にAgentCoreへ戻す。入出力契約を固定しておくことで、切り替えコストを最小化する |
| **残存リスク** | Bedrock 直接呼び出しへ一時退避した場合、AgentCore の活用アピールが弱まる |
