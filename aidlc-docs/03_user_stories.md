# 03 User Stories

**プロジェクト名**: Context Butler / 説明補足AI（Explain Bot）

---

## Persona 定義

| Persona ID | 名前 | 役割 | 主な課題 |
|-----------|------|------|---------|
| P1 | 途中参加の開発者 | プロジェクト中途参加のエンジニア | 過去の経緯・意思決定の背景が分からない |
| P2 | 新人メンバー | 入社 3 ヶ月以内のメンバー | チームの暗黙知・略語・固有用語が分からない |
| P3 | 非エンジニアのステークホルダー | 営業・企画・情シスのメンバー | 技術的な投稿の意味が分からない |
| P4 | 説明が雑な投稿者 | 忙しいシニアエンジニア | 丁寧に説明する時間がない |
| P5 | 忙しいマネージャー | チームリーダー・PM | 追加質問する時間がない。スレッドを追う時間がない |

## User Stories

### P1: 途中参加の開発者

---

**US-01** （優先度: Must）

```
As a 途中参加のプロジェクトメンバー,
I want to 分かりにくい Slack 投稿の三点リーダから補足説明を呼び出せる,
So that 追加質問せずに投稿の背景・前提・次アクションを理解できる.
```

**Acceptance Criteria**:
- [ ] Slack の三点リーダメニューに「Explain with 説明補足AI」が表示される
- [ ] 選択後 3 秒以内に何らかの応答がある（ack または「生成中です」メッセージ）
- [ ] 元投稿のスレッドに補足文が返信される
- [ ] 補足文に「背景」「前提」「次アクション」が含まれる
- [ ] 補足文の冒頭が「補足です。」で始まる

---

**US-02** （優先度: Must）

```
As a 途中参加のプロジェクトメンバー,
I want to 補足文に関連 Issue や議事録への言及が含まれる,
So that 自分で資料を探す手間を省ける.
```

**Acceptance Criteria**:
- [ ] 文脈取得 Agent が Bedrock Knowledge Bases から関連情報を検索する
- [ ] 取得した情報の出典（KB / Drive / GitHub）が補足文に反映される
- [ ] 取得できなかった情報は「この投稿だけでは明記されていません」と明示される

---

### P2: 新人メンバー

---

**US-03** （優先度: Must）

```
As a 新人メンバー,
I want to 補足文に略語・専門用語の説明が含まれる,
So that チームの暗黙知を追加質問なしに理解できる.
```

**Acceptance Criteria**:
- [ ] 省略抽出 Agent が略語・専門用語を implicit_knowledge として抽出する
- [ ] 補足文生成 Agent が用語説明を補足文に追加する
- [ ] 用語説明が過剰にならない（必要なものだけ含める）

---

**US-04** （優先度: Should）

```
As a 新人メンバー,
I want to 補足レベルを「かんたん」に設定できる,
So that 専門用語を避けた平易な補足文を受け取れる.
```

**Acceptance Criteria**:
- [ ] Slack モーダルで補足レベル（かんたん / 標準 / 詳細）を選択できる
- [ ] 「かんたん」を選択した場合、補足文が平易な言葉で書かれる
- [ ] リテラシーレビュー Agent が literacy_level に応じて文体を調整する

---

### P3: 非エンジニアのステークホルダー

---

**US-05** （優先度: Must）

```
As a 非エンジニアのステークホルダー,
I want to 技術的な投稿の意味を補足文で理解できる,
So that エンジニアに追加質問しなくても投稿の意図が分かる.
```

**Acceptance Criteria**:
- [ ] 補足文に技術用語の説明が含まれる
- [ ] 補足文が非エンジニアにも理解できる言葉で書かれる
- [ ] 「次に取るべき行動」が具体的に書かれる（誰が何をすべきか）

---

**US-06** （優先度: Should）

```
As a 非エンジニアのステークホルダー,
I want to 想定読者を「非エンジニア」に設定できる,
So that 技術的な詳細を省いた補足文を受け取れる.
```

**Acceptance Criteria**:
- [ ] Slack モーダルで想定読者（新人 / 非エンジニア / エンジニア / プロジェクトメンバー）を選択できる
- [ ] 「非エンジニア」を選択した場合、技術用語に説明が追加される
- [ ] リテラシーレビュー Agent が audience_type に応じて文体を調整する

---

### P4: 説明が雑な投稿者

---

**US-07** （優先度: Must）

```
As a 説明が雑な投稿者,
I want to 雑に投稿しても AI が補足してくれる,
So that 毎回丁寧に背景を書く手間を省ける.
```

**Acceptance Criteria**:
- [ ] 背景・前提・次アクションが省略された投稿でも補足文が生成される
- [ ] 補足文が発言者の意図を変えていない
- [ ] 補足文が「補足です。」で始まり、元投稿を批判しない表現になっている

---

**US-08** （優先度: Must）

```
As a 説明が雑な投稿者,
I want to 補足文が事実と推測を分けて書かれている,
So that 自分が言っていないことを断定されない.
```

**Acceptance Criteria**:
- [ ] 取得した文脈から確認できることは断定形で書かれる
- [ ] 確認できないことは「〜と考えられます」「〜の可能性があります」と書かれる
- [ ] 不明な点は「この投稿だけでは明記されていません」と明示される
- [ ] リテラシーレビュー Agent が根拠のない断定を検出・修正する

---

### P5: 忙しいマネージャー

---

**US-09** （優先度: Must）

```
As a 忙しいマネージャー,
I want to 補足文に「次に取るべき行動」が明確に書かれている,
So that スレッドを全部読まなくても誰が何をすべきか分かる.
```

**Acceptance Criteria**:
- [ ] 補足文に「次に取るべき行動」セクションが含まれる
- [ ] 誰が何をすべきかが具体的に書かれる
- [ ] 期限がある場合は明記される

---

**US-10** （優先度: Must）

```
As a 忙しいマネージャー,
I want to 補足文が適切な長さに収まっている,
So that 長すぎる補足文を読む時間を取られない.
```

**Acceptance Criteria**:
- [ ] 補足文が 200〜500 文字程度に収まる（目安）
- [ ] リテラシーレビュー Agent が補足過多をチェックする
- [ ] 不要な情報が追加されていない

## MVP で実装する User Story

MVP（予選デモ）では以下の Must ストーリーを実装します。

- US-01: 三点リーダから補足を呼び出せる
- US-02: 関連情報が補足文に反映される（KB 連携）
- US-03: 略語・専門用語の説明が含まれる
- US-05: 技術的な投稿の意味を理解できる
- US-07: 雑な投稿でも補足文が生成される
- US-08: 事実と推測が分けて書かれる
- US-09: 次に取るべき行動が明確に書かれる
- US-10: 補足文が適切な長さに収まる

Should ストーリー（US-04, US-06）は予選後に追加します。
