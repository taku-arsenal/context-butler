# 01 Business Intent

**プロジェクト名**: Context Butler / 説明補足AI（Explain Bot）  
**ハッカソン**: AWS Summit Japan 2026 AI-DLC ハッカソン  
**テーマ**: 「人をダメにする」

---

## Business Intent（一文）

> Slack 上の説明不足な投稿を、受け手が三点リーダから呼び出すだけで、AI が背景・前提・用語・判断理由・次アクションを補足してスレッドへ返信する。

---

## ハッカソンテーマとの接続

テーマ「人をダメにする」に対して、本プロダクトは **「説明責任・文脈整理・聞き手への配慮」を AI に外注する** という切り口で接続します。

通常、人は他者に伝わるように説明するために以下の努力をしています。

| 人間がやっていること | Context Butler が代わりにやること |
|---------------------|----------------------------------|
| 背景を整理する | 省略抽出 Agent が背景の欠如を検出し、文脈取得 Agent が補完する |
| 前提条件を書く | 補足文生成 Agent が前提を明示する |
| 関連資料を探す | 文脈取得 Agent が KB・Drive・GitHub から自動収集する |
| 専門用語を噛み砕く | 補足文生成 Agent が用語説明を追加する |
| 聞き手の知識レベルを考慮する | リテラシーレビュー Agent が読み手に合わせて調整する |
| 誤解が起きないように書く | リテラシーレビュー Agent が事実と推測を分離する |
| 必要な行動を明示する | 補足文生成 Agent が次アクションを整理する |

発言者は雑に投稿しても AI が補足し、聞き手は追加質問を減らせます。**人間の「説明する力」を弱らせ、人をダメにする** — しかし業務効率化としては普通に便利で、実用性もある。そのギリギリのラインを攻めたプロダクトです。

---

## 対象ユーザー

### プライマリユーザー（補足を受け取る側）

| ペルソナ | 課題 |
|---------|------|
| プロジェクト途中参加者 | 過去の経緯・意思決定の背景が分からない |
| 新人メンバー | チームの暗黙知・略語・固有用語が分からない |
| 非エンジニアのステークホルダー | 技術的な投稿の意味が分からない |
| 忙しいマネージャー | 追加質問する時間がない |

### セカンダリユーザー（投稿する側）

| ペルソナ | 課題 |
|---------|------|
| 説明が雑な投稿者 | 丁寧に説明する時間がない、または説明が苦手 |
| 専門知識が深いエンジニア | 自分には自明なことを省いて投稿してしまう |

---

## 解決する業務課題

業務チャットでは、発言者が背景・前提・判断理由を省いたまま投稿し、受け手が確認質問や認識合わせに時間を使うことが繰り返されています。

**典型的な問題投稿**:
```
「例の認証の件、来週から切り替える方向で。影響あるところだけ確認お願いします。」
```

この投稿には以下が不足しています。

- 「例の認証」とは何か（どのシステムの話か）
- なぜ切り替えるのか（判断理由）
- いつ決まったのか（意思決定の経緯）
- 誰が何を確認すべきか（アクション対象者）
- 影響範囲はどこか
- 関連する Issue / 資料はどこか

受け手は「これは何の話？」「背景は？」「何をすればいい？」と追加質問するか、自分で文脈を探すしかありません。

**Context Butler が生成する補足文の例**:
```
補足です。この投稿は、社内ポータルの認証方式を既存の独自認証から
Cognito 連携に切り替える件についての共有です。

背景として、先週の定例でセキュリティ運用負荷を下げるため、認証基盤を
AWS 側に寄せる方針が合意されています。対象はまず開発環境で、本番切り替えは
別途判断予定です。

確認すべき人は、ログイン処理・ユーザー管理・権限チェックに関わる実装を
持つ担当者です。

次のアクションは、関連 Issue に影響範囲を追記し、来週の切り替え前に
懸念点をこのスレッドへ返信することです。

なお、本番適用日とロールバック手順はこの投稿だけでは明記されていません。
```

## MVP で証明したい仮説

| 仮説 | 検証方法 |
|------|---------|
| 説明不足な投稿でも、AI の補足があれば受け手が追加質問なしに意図・背景・次アクションを理解できる | デモ参加者・審査員が補足文を読んで元投稿の意図を説明できるか確認する |
| Slack の三点リーダという自然な UX で、業務フローを壊さずに補足を呼び出せる | デモで実際に操作して確認する |
| AgentCore Runtime + Strands Orchestrator を第一候補として 4 Agent ステージを順次制御し、30 秒以内の補足文生成を目指せる | 処理時間を計測する。詰まった場合も同じ入出力契約で Bedrock 直接呼び出しに退避できることを確認する |
| 事前に用意したテストデータの想定補足ポイントを 80% 以上満たす補足文を生成できる | テストデータに対して充足率を採点する |

## なぜ今作るのか

- **Amazon Bedrock AgentCore Runtime** が 2025 年に GA し、マルチエージェント構成を AWS 上で本番運用できる環境が整った
- **Strands Agents** により Python ベースで Agent パイプラインを簡潔に実装できるようになった
- リモートワーク・非同期コミュニケーションの普及により、Slack 上の短い投稿が増加し、文脈不足による追加質問コストが顕在化している
- AWS Summit Japan 2026 AI-DLC ハッカソンが、AgentCore / MCP / A2A を活用した実用的な AI アプリを評価する場として最適なタイミング
